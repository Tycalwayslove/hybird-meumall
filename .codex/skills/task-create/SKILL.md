# Task Create

## 何时使用

当用户要求创建、定义、拆分、登记一个新的 AI 任务，或用自然语言描述“想做一件事”并希望 AI 汇总成任务时使用。

本 skill 只做任务澄清、任务草案和任务文档创建，不实现业务代码。

## 核心原则

- 用户可以用自然语言描述需求，不需要一次性写完整任务。
- 信息不足时，不要立即创建任务文件。
- 每轮最多问 3 个关键问题。
- 信息足够后，先输出任务草案并请求确认。
- 只有用户明确确认后，才创建 `.ai/tasks/*.md` 并更新 `.ai/TODO.md`。

## 输入

- 用户自然语言需求、产品想法、缺陷描述或技术改造意图。
- `AGENTS.md`。
- `.ai/AI_CONTEXT.md`、`.ai/PROJECT_STATE.md`、`.ai/TODO.md`。
- `docs/07_AI_WORKFLOW.md`。
- 与任务领域相关的 `docs/*.md`。

## 对话式创建流程

### 阶段一：理解意图

1. 读取必要上下文：
   - `AGENTS.md`
   - `.ai/AI_CONTEXT.md`
   - `.ai/PROJECT_STATE.md`
   - `.ai/TODO.md`
   - `docs/07_AI_WORKFLOW.md`
2. 用一句话复述用户想做什么。
3. 判断任务涉及的标签：
   - `docs`
   - `bridge`
   - `release`
   - `theme`
   - `api`
   - `ui`
   - `infra`
4. 判断信息是否足够创建任务。

### 阶段二：多轮澄清

如果信息不足，先提问，不落盘。

优先澄清：

- 目标：这个任务最终要解决什么问题？
- 范围：本次包含哪些内容？
- 不包含范围：哪些内容明确不做？
- 验收标准：做到什么程度算完成？
- 验证方式：用测试、构建、人工检查，还是其他方式验证？
- 风险和假设：是否影响原生兼容、发布、回滚、主题、API 或用户数据？
- 文件范围：是否已有明确要新增或修改的文件？

提问规则：

- 每轮最多 3 个问题。
- 优先问会影响范围和验收标准的问题。
- 能从项目文档推断的内容不要反复问，但要在草案中标为“假设”。
- 影响发布安全、原生兼容或用户数据的问题必须明确确认。

### 阶段三：生成任务草案

信息足够后，先输出任务草案，不创建文件。

草案必须包含：

- 任务标题
- 目标
- 标签
- 包含范围
- 不包含范围
- 相关文档
- 相关文件
- 验收标准
- 验证方式
- 风险和假设
- 仍需确认的问题，如果有

草案后必须询问用户是否确认创建任务。

可以使用这样的确认句：

> 以上是任务草案。确认后我会创建 `.ai/tasks/YYYY-MM-DD-short-title.md` 并更新 `.ai/TODO.md`。是否确认创建？

### 阶段四：确认后落盘

只有用户明确表达以下含义时，才可以创建任务文件：

- “确认”
- “创建任务”
- “可以落盘”
- “按这个创建”
- “没问题，生成任务”

落盘动作：

1. 在 `.ai/tasks/YYYY-MM-DD-short-title.md` 创建任务文件。
2. 更新 `.ai/TODO.md`。
3. 必要时更新 `.ai/CHANGE_SUMMARY.md`。
4. 输出创建的文件路径和后续建议。

## 是否可以落盘

### 可以落盘

- 用户已明确确认。
- 目标、范围、验收标准和验证方式都清楚。
- 不存在影响架构、发布、原生兼容或用户数据的未决关键问题。
- 不包含未经确认的业务实现。

### 不能落盘

- 用户只是描述想法，还没有确认创建。
- 范围边界不清楚。
- 验收标准不清楚。
- 验证方式不清楚。
- 存在关键技术决策未确认。
- 任务看起来混合了多个独立子任务，需要先拆分。

## 任务文件模板

```markdown
# 任务：<短标题>

## 目标

<本任务要完成什么。>

## 标签

- <scope label>

## 范围

### 包含

- 

### 不包含

- 未明确要求时不实现业务功能。

## 上下文

- 相关文档：
- 相关文件：

## 验收标准

- [ ] 

## 验证

- 命令或人工检查：
- 预期结果：

## 风险和假设

- 
```

## 输出

未确认前：

- 澄清问题，或任务草案。
- 不创建文件。

确认后：

- `.ai/tasks/` 下的任务文件。
- 更新后的 `.ai/TODO.md`。
- 必要时更新 `.ai/CHANGE_SUMMARY.md`。
- 简短说明任务已创建及下一步建议。

## 示例

用户：

> 使用 task-create：我想做一个 manifest 版本解析能力，支持稳定版、灰度版、强制版和回滚。

AI 应先问：

> 我理解这是一个 `release/infra` 任务：为 manifest 增加版本解析能力。创建任务前需要确认 3 点：
> 1. `forceVersion` 的优先级是否高于 `rollbackVersion`？
> 2. 灰度规则第一版只按 `userId` 命中，还是也需要 `platform/appVersion`？
> 3. 本任务是否需要同时引入测试运行器？

不能直接创建任务文件，除非用户已经确认草案。
